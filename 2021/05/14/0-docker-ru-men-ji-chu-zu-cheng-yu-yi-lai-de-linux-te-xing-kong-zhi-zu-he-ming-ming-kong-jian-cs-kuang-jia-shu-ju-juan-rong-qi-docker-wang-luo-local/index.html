0 容器和虚拟机的区别

容器是一种虚拟化的方案，
虚拟机，通过中间层将多台独立的机器虚拟运行在物理硬件上;
容器，是直接运行在操作系统内核之上的用户空间，也称操作系统虚拟化。

容器只能运行在相似的操作系统上，docker依赖linux内核特性Namespace和Cgroups（Control Group）

0.1 部署应用时

虚拟机 包含：app—lib—OS（几十G，模拟硬件行为，磁盘和内存消耗大）
容器 包含：app—lib（小，但依赖内核特性 难安装 难管理 不易自动化）
因此docker应运而生。。。go语言编写，13年初发布，Apache2.0开源协议

1 docker基本组成（client–daemon–container–registy）

把应用程序自动部署到容器的开源引擎。

开发人员只需关注app，运维人员只需关注docker管理，开发 测试 生产部署环境一致性。
推荐一个docker容器对应一个app，高内聚 低耦合 单一任务

docker由4部分组成：
客户端—守护进程—镜像—容器—仓库
client—daemon—image—container—registry

组成，**通过 客户端 访问守护进程 进而 操作 容器，容器通过镜像创建，镜像保存在仓库中。**

1.1 守护进程

1.1.1 C/S框架

客户端/服务器 框架，对应于docker的 docker客户端/守护进程
客户端向服务端发出请求，服务端处理完所有工作并返回结果。
支持本地和远程访问。

1.1.2 C/S运行模式深入

在shell 或者 bash中用cmd与docker守护进程链接，执行完后守护进程返回结果。这是client与docker通信的主要的方式。
之外还提供Remote API，便于自己的程序与docker集成。是RESTful风格的API，与一般网络通信接口差不多。

RESTFUL是一种网络应用程序的设计风格和开发方式



客户端与守护进程链接为socket，方式三种，默认unix，

在服务端配置docker守护进程的启动选项，实现客户端远程访问docker服务，命令参数-H。
也可以在客户端添加环境变量DOCKER_HOST实现，置空即可链接本地docker服务。

1.3 docker容器

容器通过镜像启动，是docker的执行单元，容器中可以运行多个app，
镜像是docker的构建和打包阶段，容器是启动和执行阶段。

2 docker依赖的linux特性

命名空间（namespace） 和 控制组（Control group）

2.1 命名空间

编程语言中，封装—>代码的隔离
操作系统中，系统资源的隔离（进程 FS 网络 。。），目的之一是实现轻量级虚拟化服务。

同命名空间下进程相互感知，不同命名空间下进程独立/隔离。

2.1.1 docker提供5种命名空间：

  • PID（process ID）进程
  • NET（network）网络
  • IPC 管理进程间通信访问
  • MNT（mount）管理挂在点
  • UTS（Unix Timesharing System）隔离内核和版本标识

这些隔离的资源通过控制组管理。

2.2 控制组

限制 记录 隔离 进程组使用物理资源的机制。
来源于google，linux kernel 2.6.24@2007 整合该机制。

cgroups为容器而生。

  • 资源限制（内存等）
  • 优先级设定
  • 资源计量
  • 进程控制（挂起 恢复）

3 docker容器的能力

  • 文件系统的隔离（每个容器都有自己的root FS）
  • 进程的隔离（每个容器都在自己的进程环境中）
  • 网络隔离（容器间虚拟网络接口 和 IP地址分开）
  • 资源的隔离于分组（Cgroups将cpu 内存等资源独立分配给容器）

4 docker深入原理

4.1 构建产生中间层镜像缓存，调试镜像构建

基于Dockerfile构建镜像的过程：

1 从基础镜像运行一个容器
2 执行一条指令,对容器做出修改
3 执行类似 docker commit的操作,提交一个新的镜像层
4 再基于刚提交的镜像运行一个新容器
5 执行 Dockerfile中的下一条指令,直至所有指令执行完毕
不断删除容器保存镜像（中间层镜像，cache中，构建缓存），因此可以跟踪调试镜像的构建过程。可以通过docker build --no-cache不使用缓存，或 dockerfile中添加ENV REFRESH_DATE 2021-05-04，后面命令便会刷新缓存。

4.3 数据卷

4.3.1 什么是数据卷( Data volume)：

  • 数据卷是经过特殊设计的目录,可以绕过联合文件系统(UFS),为一个或多个容器提供访问数据卷设计的目的,
  • 在于数据的永久化,它完全独立与容器的生存周期,因此,Docker不会在容器删除时删除其挂载的数据卷,也不会存在类似的垃圾收集机制,对容器引用的数据卷进行处理。

向镜像中添加卷，一个卷可以存在于一个或多个容器的特定目录，绕过联合文件系统，提供共享数据 数据持久化功能。独立于容器的生命周期，存在于宿主机中。

4.3.2 数据卷( Data Volume)的特点

  • 数据卷在容器启动时初始化,如果容器使用的镜像在挂载点包含了数据,这些数据会拷贝到新初始化的数据卷中
  • 数据卷可以在容器之间共享和重用
  • 可以对数据卷里的内容直接进行修改
  • 数据卷的变化不会影响镜像的更新
  • 卷会一直存在,即使挂载数据卷的容器已经被删除
