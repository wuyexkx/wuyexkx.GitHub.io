要运行容器，需要镜像，docker有三种方法生成镜像：

  • 写dockerfile（自建，需要学习写dockerfile，但通用，推荐）
  • 从仓库pull（远程，简单但难以维护，不推荐）
  • 从tar文件load（从本地load）

2.1 从远程仓库pull现成的最新ubantu镜像

# 下载现有最新ubuntu镜像
$ docker pull ubuntu
# 查看本地目前有哪些镜像
$ docker images

hello-world   latest    d1165f221234   8 weeks ago   13.3kB
# 以该镜像为基础构建新的容器（run: Run a command in a new container）
$ docker run -it 7e /bin/bash   #/bin/bash现在可以不加，默认运行。早期是必须有个进程运行#

# 查看安装的版本
$ cat /etc/issue

2.1.2 修改镜像后保存为自己的镜像

若用commit，会使得镜像越来越臃肿，而且难以追溯制作人的操作。

使用docker commit命令虽然可以比较直观的帮助理解镜像分层存储的概念，但是实际环境中并不会这样使用。
https://blog.csdn.net/qq_37674858/article/details/81743635
docker commit有一些特殊的应用场合，比如被入侵后保存现场等。但是，不要使用docker commmit定制镜像，定制行为应该使用Dockerfile来完成。

$ docker commit [OPTIONS] CONTAINER [REPOSITORY[:TAG]]
$ docker commit -m "messages" --author='xxx' CONTAINER [REPOSITORY[:TAG]]
docker commit ubuntu20 ubuntu20-t

2.2 通过dockerfile构建镜像

2.2.1 一个简单例子，编写dockerfile

Dockerfile文件位置不重要
比如Dockerfile/Dockerfile1/Dockerfile，内容如下：

# first dickerfile
FROM ubuntu:20.04
MAINTAINER wuyexkx "111@qq.com"
RUN apt update
RUN apt install -y nginx
EXPOSE 80

构建镜像build， -t image名称

docker build -t "image-t1" DOCKERFILE_PATH

2.2.2 基于Dockerfile构建镜像，指令和原理

基于Dockerfile构建镜像的过程：

1 从基础镜像运行一个容器
2 执行一条指令,对容器做出修改
3 执行类似 docker commit的操作,提交一个新的镜像层
4 再基于刚提交的镜像运行一个新容器
5 执行 Dockerfile中的下一条指令,直至所有指令执行完毕
不断删除容器保存镜像（中间层镜像，cache中，构建缓存），因此可以跟踪调试镜像的构建过程。可以通过docker build --no-cache不使用缓存，或 dockerfile中添加ENV REFRESH_DATE 2021-05-04，后面命令便会刷新缓存。

# 镜像的命令
FROM <image> or <image:tag> image为基础镜像
MAINTAINER 					作者  联系方式
RUN	<command>				构建镜像时运行的指令，层级构建，RUN分级构建，每RUN生成一个中间件
EXPOSE <port>				告知镜像暴露的端口，但run时需要再添加-p指定（安全考虑）

# 容器运行时执行的命令
CMD	<command>				启动容器时运行的指令，会被run中的args覆盖 如/bin/bash	
ENTRYPOINT <command>		与CMD类似，不会被覆盖且不执行args指定的app，若要强制覆盖加 --entrypoint参数
							可以用ENTRYPOINT指定命令 CMD指定默认参数（run时可以覆盖参数而不修改命令）
	
# 设置镜像的目录和文件				
ADD <src> <desc>			将文件或目录复制到构建的镜像中，src可以是相对，desc必须是镜像中的绝对path
							ADD ["src" "desc"] 适用于路径中有空格的情况
COPY 						与ADD类似，但ADD包含tar解压缩功能，COPY单纯复制
VOLUME						向镜像中添加卷，一个卷可以存在于一个或多个容器的特定目录，绕过联合文件系统，提供共享数据 数据持久化功能

# 容器运行时环境设置
WORKDIR	<path>				从镜像启动容器时 在容器内部设置新的工作目录，使用绝对路径，否则传递
ENV	<key> <value>			环境变量，构建和运行有效				
USER <user>					指定用户运行，默认root

# 类似触发器的指令
ONBUILD [INSTRUCTION]		构建本镜像不执行，当本镜像本用作其他镜像的基础镜像时，触发执行命令

2.3 一个较适用的dockerfile例子

Dockerfile中使用换行文本，在末尾加 \n\ 即可。

FROM ubuntu:20.04
LABEL maintainer="1191468572@qq.com"

#定义时区参数, 避免时区设置交互, ubuntu bug
ENV TZ 'Asia/Shanghai"'
RUN echo $TZ > /etc/timezone && \
apt-get update && apt-get install -y tzdata && \
rm /etc/localtime && \
ln -snf /usr/share/zoneinfo/$TZ /etc/localtime && \
dpkg-reconfigure -f noninteractive tzdata && \
apt-get clean

# 更新源
RUN echo "deb http://mirrors.aliyun.com/ubuntu/ focal main restricted universe multiverse \n\
deb-src http://mirrors.aliyun.com/ubuntu/ focal main restricted universe multiverse \n\
\n\
deb http://mirrors.aliyun.com/ubuntu/ focal-security main restricted universe multiverse \n\
deb-src http://mirrors.aliyun.com/ubuntu/ focal-security main restricted universe multiverse \n\
\n\
deb http://mirrors.aliyun.com/ubuntu/ focal-updates main restricted universe multiverse \n\
deb-src http://mirrors.aliyun.com/ubuntu/ focal-updates main restricted universe multiverse \n\
\n\
deb http://mirrors.aliyun.com/ubuntu/ focal-proposed main restricted universe multiverse \n\
deb-src http://mirrors.aliyun.com/ubuntu/ focal-proposed main restricted universe multiverse \n\
\n\
deb http://mirrors.aliyun.com/ubuntu/ focal-backports main restricted universe multiverse \n\
deb-src http://mirrors.aliyun.com/ubuntu/ focal-backports main restricted universe multiverse">/etc/apt/sources.list

# 更新和升级
RUN apt update -y && apt upgrade -y
RUN apt install net-tools inetutils-ping openssh-server samba samba-common git vim curl -y
RUN apt install nodejs npm -y

3 docker常用命令

镜像的拉查建跑删：
docker pull | images | build | run | rmi -f，要删images(镜像) 先删对应container(容器)
容器查停跑删：
docker ps -a | stop | exec | rm

交互式容器

默认一次执行后结束，
启动交互式容器，进入bash：
docker run -i -t IMAGE /bin/bash

查看容器信息：
docker inspect xxx

重命名容器：

docker rename [旧容器名] [新容器名]

创建容器时设定容器主机名：

docker run -it -h [主机名] IMAGE

创建时设定容器名称：

docker run -it --name [容器名] IMAGE

创建容器时设定主机名 容器名：

docker -it --name [容器名] -h [主机名] IMAGE

守护式容器

1 从交互式中切到守护式后台运行
ctlr+P ctrl+Q
2 贴附到运行中的容器（转入到容器），
docker attach IAMGE
3 直接启动守护式
docker run -d IMAGE

查看容器内部运行情况

1 日志：
docker logs [-f] [-t] [--tail] IMAGE

2 查看哪些容器在运行docker ps
3 查看容器内部运行的进程docker top

在运行的容器内启动新进程：exec

运行已经退出的容器

docker start -i IMAGE

停止容器

docker stop IMAGE   发信号死掉
docker kill IMAGE   直接死掉

docker镜像信息

docker images --no-trunc     id不截断显示  《truncation截断》

与registry（docker hub）的互动

搜索仓库

为容器添加数据卷

在docker inspect 容器名 都可以看到详细信息

1 容器的数据卷

-v 宿主机文件路径:容器文件绝对路径

sudo docker run -v ~/xxxx:/xxx -it IMAGE /bin/bash  # 若本机无该目录则自动创建
sudo docker run -v ~/xxxx:/xxx:ro -it IMAGE /bin/bash # 设置卷的权限只读

与docker run命令不同，通过Dockerfile的VOLUME创建的数据卷时，不能映射本地已经的目的中，并且启动不同容器名的容器映射路径也不一样。因此没办法在容器间数据共享—>数据卷容器应运而生。

2 数据卷容器

什么是数据卷容器

命名的容器挂载数据卷，其他容器通过挂载这个容器实现数据共享，挂载数据卷的容器，就叫做数据卷容器。

数据容器 容器 宿主机之间的关系

有数据卷的容器 与 宿主机 通过数据卷共享数据，其他容器链接该容器完成数据共享。

docker run --volume-from 已挂载数据卷的容器名

多租户的服务器中非常使用，多个容器共享目录，不暴露服务器的实际目录情况。

数据卷容器 数据备份 还原

通过run一个新的容器将数据卷容器挂载过来，然后执行本分或者还原操作。


修改docker配置文件，加速下载库

vim /etc/default/docker
添加
DOCKER_OPTS="xxx"
重启docker的守护进程
sudo service docker restart

4 权限问题 systemctl

在在启动容器后对容器进行配置，修改一个参数时发现权限和init进程没有启动的问题。

提示：
System has not been booted with systemd as init system (PID 1). Can't operate. Failed to connect to bus: Host is down

创建容器时添加--privileged 选项 和 启动 第一个进程/sbin/init：

docker run -itd --name=[容器名] -h [主机名] --privileged=true [镜像名] /sbin/init

然后：
docker exec -it [容器名] /bin/bash

仅首次有效不能永久解决，这样只能保证首个进程是/sbin/init，而非systemd
