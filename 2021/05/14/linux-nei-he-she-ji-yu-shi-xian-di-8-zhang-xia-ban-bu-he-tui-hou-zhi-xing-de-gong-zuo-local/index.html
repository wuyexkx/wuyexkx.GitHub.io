Linux内核设计与实现 第8章 下半部和推后执行的工作

Linux内核设计与实现 陈莉君（第三版）
Linux内核设计与实现 陈莉君（第三版）
发布日期:   2021-05-14
文章字数:   4.6k

@[toc]
操作系统必须有一个快速、异步、简单的机制负责对硬件做出迅速响应并完成那些时间要求很严格的操作。（上半部）
对于那些其他的、对时间要求相对宽松的任务，就应该推后到中断被激活以后再去执行。（下半部）

硬件资源异步向处理器发信号，中断就是由硬件来打断操作系统。
中断可以打断进程、内核本身、其他中断程序的代码执行。

中断处理程序本身存在一定局限，所以只能完成 整个中断处理流程 的上半部分，因为：

  1. 中断是异步执行的，打断了其他代码，所以中断处理程序执行要越快越好；
  2. 中断处理程序在执行时，可能会屏蔽掉其他硬件中断，导致其他硬件无法与操作系统通信，所以中断处理程序应该越快越好；
  3. 中断处理程序往往需要对硬件进行操作，所以有很高的时限要求；
  4. 中断处理程序不在进程上下文中，不能阻塞。

    下半部

    中断处理程序会异步执行，并且在最好的情况下也会锁定当前的中断线。当前中断线在所有处理器上都会被屏蔽。最坏的情况还会禁止本地所有中断。

上下半部并没有绝对的界限，通常考虑放在上半部的任务：

  1. 任务对时间非常敏感
  2. 任务与硬件相关
  3. 任务保证不会被打断，包括不被其他中断打断

其他所有任务，考虑放在下半部执行。下半部执行的关键是在它们运行的时候，允许响应所有中断。

下半部的环境

上半部只能通过中断处理程序实现。而下半部可以通过多种机制实现。
用来实现下半部的各种机制分别由不同的接口和子系统组成。

内核提供三种不同形式的下半部实现机制，软中断、tasklet、工作队列。可以把下半部任务推迟到除了现在以外的任何时刻执行。tasklet本身是通过软中断实现的。
而 内核定时器 把下半部任务推迟到某个确定的时间段之后执行。

软中断

软中断使用较少，而tasklet是下半部的常用形式。但是tasklet是由软中断实现，所以先研究软中断。

软中断的实现

软中断是在编译期间静态分配的，tasklet能被动态的注册或注销。软中断由softirq_action结构表示，定义在linux/interrupt.h中，

struct softirq_action {
	void (*action)(struct softirq_action *);
};

在kernel/softirq.c中定义了一个包含有32个该结构体的数组，每个被注册的软中断都占据其中一项，最多有32个软中断，32为定值不能动态改变，在版本2.6中只用到了9个。

static struct softirq_action softirq_vec[NR_SOFTIRQS];

软中断处理程序

软中断处理程序原型。当内核运行一个软中断程序时，就执行这个action函数，唯一参数为struct softirq_action *型，调用方式softirq_action ->action(my_softirq)。

void softirq_handler(struct softirq_action *);

一个软中断不会抢占另一个软中断，
能抢占软中断的只有中断处理程序。
软中断可以在其他处理器上同时执行。

执行软中断

一个注册的软中断必须再被标记后才会执行，称触发软中断（raising the softirq）。
通常中断处理程序返回前标记它的软中断，使其在稍后被执行。
在以下地方软中断会被检查执行：

  1. 从一个硬件中断代码处返回时；
  2. 在ksoftirqd内核线程中；
  3. 在那些显示检查和执行待处理的软中断的代码中，如网络子系统。

最终软中断都要在do_softirq()中执行。do_softirq()会循环遍历调用每个软中断程序。

do_softirq()的简化代码如下。
pending保存待处理软中断32位位图（如果第n位被置1，那么第n位对应类型的软中断等待处理）；
位图已保存，清理实际软中断位图set_softirq_pending(0);
依次执行被标记的软中断，最多执行32个。

u32 pending;
pending = local_softirq_pending();
if (pending) {
	struct softirq_action *h;
	
	/* reset the pending bitmask */
	set_softirq_pending(0);
	
	h = softirq_vec;
	do {
		if (pending & 1)
		h->action(h);
		h++;
		pending >>= 1;
	} while (pending);
}

使用软中断

目前只有两个子系统直接使用软中断，网络和SCSI。内核定时器和tasklet都是建立在软中断上的。

优先使用tasklet，可以动态生成，对加锁要求不高，使用方便，性能不错。
当然对于时间要求严格并能自己高效地完成加锁工作的应用，软中断会是正确的选择。

分配索引

编译期间，通过在linux/interrupt.h中定义一个枚举类型来静态的声明软中断，内核用从0开始的索引表示相对优先级。索引号小的软中断优先执行。建立新的软中断必须在此枚举类型中加入新的项。不是简单的放在末尾。

注册软中断处理程序

调用open_softirq()注册软中断处理程序，参数为软中断的索引号 和 处理函数。如网络子系统，在net/core/dev.c中。

open_softirq(NET_TX_SOFTIRQ, net_tx_action);
open_softirq(NET_RX_SOFTIRQ, net_rx_action);

软中断处理程序允许响应中断，但自己不能休眠。
软中断处理程序运行时，当前中断处理器上的软中断被禁止，其他处理器可以同时运行处理程序。
如果同一软中断在执行时 同时再次被触发，那么另一个处理器可以同时运行其处理程序。也就是说任何共享数据都需要严格的锁保护。

也就是说同一软中断程序可能在多个处理器上运行，共享数据需要严格的锁保护。单纯的禁止软中断程序同时执行不是很理想，通过互斥锁来防止并发执行那就没有软中断的必要了。因此，大部分软中断处理程序都通过单处理器数据（只属于单个处理器的数据）或者其他技巧来避免显示的加锁，从而提供更出色的性能。
而tasklet的同一个处理程序的多个实例不能在多个处理器上同时运行。

触发软中断

通过在枚举列表中添加新项 以及 调用open_softirq()进行注册以后，就要触发该软中断，raise_softirq()将软中断设置为挂起状态，下次调用do_softirq()函数时投入运行。

raise_softirq()在触发软中断之前要禁止中断，出发后再恢复中断状态。如果中断本来就禁止，那么可以调用raise_softirq_irqoff()。

在中断处理程序中触发软中断是常用方式。

tasklet

利用软中断实现的中断下半部机制，和进程没有任何关联。
tasklet与软中断相似，但接口简单，锁保护要求更低。 执行频率很高连续性要求很高的情况才需要软中断，否则就用tasklet。

tasklet的实现

tasklet_struct结构体

tasklet由tasklet_struct结构表示，在linux/interrupt.h中定义，

struct tasklet_struct {
	struct tasklet_struct *next; /* next tasklet in the list */
	unsigned long state; 		 /* state of the tasklet */
	atomic_t count; 			 /* reference counter */
	void (*func)(unsigned long); /* tasklet handler function */
	unsigned long data; 		 /* argument to the tasklet function */
};

fun成员是tasklet的处理程序（像软中断中的action一样）。

state成员只能在0、TASKLET_STATE_SCHED、TASKLET_STATE_RUN之间取值，
TASKLET_STATE_SCHED表明tasklet已被调度，正准备投入运行，
TASKLET_STATE_RUN表明该tasklet正在运行，只有在多处理器的系统上才会作为一种优化来使用，单处理器系统任何时候都清楚单个tasklet是不是在运行。

count成员是tasklet的引用计数器，如果不为0则tasklet被禁止，不允许执行；为0时并且在被设置为挂起时才能够执行。

调度tasklet

已调度的tasklet（等同于被触发的软中断），存放在两个单处理器数据结构：tasklet_vec （普通tasklet）和tasklet_hi_vec（高优先级的tasklet），都是由tasklet_struct结构构成的链表，每个元素代表不同的tasklet。
tasklet由tasklet_schedule()和 tasklet_hi_schedule()函数进行调度，函数区别只在一个使用TASKLET_SOFTIRQ，另一个使用HI_SOFTIRQ。

使用tasklet

声明自己的tasklet

  1. 静态创建，用linux/interrupt.h中的宏，根据给定名称静态创建task_struct结构，当本tasklet被调度后func会执行，func的参数由data给出。两个宏的区别在于初始引用计数值不同。 
    DECLARE_TASKLET(name, func, data)
DECLARE_TASKLET_DISABLED(name, func, data);
  2. 动态创建，tasklet_init(t, tasklet_handler, dev);

编写你自己的tasklet处理程序

void tasklet_handler(unsigned long data);

因为是靠软中断实现，所以tasklet不能睡眠，意味着不能在tasklet中使用信号量或者其他阻塞式函数。
tasklet运行时允许响应中断，如果tasklet和中断处理程序共享了数据则 必须做好防护工作（如关闭中断然后获取锁）。
同一个tasklet不会同时执行，但是如果一个tasklet与另一个tasklet共享了数据，那必须适当的保护。

调度自己的tasklet

通过调用tasklet_schedule()并传入相应tasklet_struct指针，那么该tasklet就会被调用，

tasklet_schedule(&my_tasklet); /* mark my_tasklet as pending */

tasklet被调度后还没运行时，如果一个相同的tasklet又被调度，仍然只运行一次；如果已经运行了，在该处理器上会重新调度并运行。一个tasklet总在调用它的处理器上执行。

ksoftirqd 处理软中断的内核线程

每个处理器都用一组辅助处理软中断（或tasklet）的内核线程。线程名字叫ksoftirqd/n，n为处理器编号。

软中断有时被触发的频率很高，如大流量网络通信期间。也就是说，当一个软中断执行时，他可以重复触发自己以便得到再次执行，如网络子系统。这可能导致用户空间无法获得足够的处理器时间。解决办法是不会立即执行重新触发的软中断。

工作队列

下半部实现方式之工作队列，是另外一种将工作推后执行的形式。交由内核线程执行，总是运行在进程上下文中，这样就拥有进程上下文中的优势，最重要的是允许重新调度和睡眠。

如何选择软中断或者工作队列？推后执行的任务需要睡眠就工作队列，否则软中断或tasklet。

工作队列可以用内核线程替换，但是内核开发者反对创建新的内核线程。XFS文件系统就是为自己创建了两个内核线程。

工作队列是唯一能在进程上下文中运行的下半部实现机制。只有它才可以睡眠。 当需要获得大量内存、获取信号量、执行阻塞式I/O操作时都非常有用。

工作队列的实现

工作队列子系统 是一个用于创建内核线程的接口，通过它创建的内核进程 负责执行 由内核其他部分排到队列里的任务。这些内核线程称为工作者线程（worker threads）。
工作队列子系统提供了缺省的工作者线程来处理推后任务。这样，工作队列就变成了 把需要推后执行的任务交给特定的通用线程的 一种接口。

缺省的工作者线程叫events/n，n是处理器编号，每个处理器对应一个线程。
缺省的工作者线程会从多个渠道得到被推后的工作，许多内核驱动程序的下半部工作都交给缺省的工作者线程去做，除非驱动或子系统需要自己的内核线程。避免工作队列中的其他任务处于饥饿，处理器密集型或性能要求严格的任务还是建议创建自己的内核线程。

表示工作者线程的数据结构 workqueue_struct

工作者线程用workqueue_struct结构表示，

/* 外部可见的工作队列抽象是  由每个CPU的工作队列组成的数组 */
struct workqueue_struct {
	struct cpu_workqueue_struct cpu_wq[NR_CPUS]; /*处理器，工作者线程中有处理器*/
	struct list_head list;
	const char *name;
	int singlethread;
	int freezeable;
	int rt;
};

结构内部有cpu_workqueue_struct 结构组成的数组，定义在kernel/workqueue.c中，数组每一项对应系统中的一个处理器，每个处理器对应一个工作者线程，每个工作者线程对应一个这样的结构体cpu_workqueue_struct ，

struct cpu_workqueue_struct {
	spinlock_t lock; /* lock protecting this structure */
	struct list_head worklist; /* list of work */
	wait_queue_head_t more_work;
	struct work_struct *current_struct;
	struct workqueue_struct *wq; /* 处理器中有工作队列结构 关联工作队列结构*/
	task_t *thread; /* associated thread */
};

给每个线程分配一个cpu_workqueue_struct ，每个处理器都有一个workqueue_struct 。工作者线程和处理器 相互关联起来。

表示工作的数据结构 work_struct

struct work_struct {
	atomic_long_t data;
	struct list_head entry;
	work_func_t func;
};

所有工作者线程都是用普通内核线程实现，都要执行worker_thread()函数。在工作者线程初始化完成之后这个函数执行死循环并休眠。当有操作被插入到队列时线程被唤醒去执行操作，没有剩余操作时又休眠。
worker_thread()
  run_workqueue()

如驱动程序需要推后执行的工作，用work_struct 表示。内部有func指针指向一个函数，这个函数就是负责处理需要推后执行的具体任务。

使用工作队列

先看默认的events任务队列，再看创建新的工作者进程。

1. 创建推后的工作

编译时静态创建，静态的创建一个名为name，处理函数为func，参数为data的work_struct结构体。

DECLARE_WORK(name, void (*func)(void *), void *data);

运行时动态创建，

INIT_WORK(struct work_struct *work, void (*func)(void *), void *data);

2. 工作队列处理函数

这个函数会由一个工作线程执行，会运行在进程上下文中。默认情况允许响应中断，不持有任何锁，如果需要函数可以睡眠。不能访问用户空间，因为内核线程在用户空间没有相关的内存映射。
而发生系统调用时，内核会代表用户空间的进程运行，此时才能访问用户空间，只有在此时它才会映射用户空间的内存。

void work_handler(void *data);

3. 对工作进行调度

工作创建之后就可以调度了，就是把给定的工作提交给缺省的events工作线程，一旦所在处理器上的工作者线程被唤醒，就会被执行。

schedule_work(&work_struct);

如果不希望马上被执行，希望过一段时间再执行，

schedule_delayed_work(&work, delay);

4. 刷新操作

直到队列中所有对象被执行以后才返回，等待所有待处理的工作执行的时候，进入休眠，所以该函数只能在进程上下文中使用。

void flush_scheduled_work(void);

取消延迟执行的工作应该调用，

int cancel_delayed_work(struct work_struct *work);

5. 创建新的工作队列

创建一个新的任务队列和与之相关的工作者线程，调用，

struct workqueue_struct *create_workqueue(const char *name);

name为内核线程的名称，创建所有工作者线程（每个处理器都有一个）。

在创建新的工作队列时无需考虑类型，而调度执行就有区别了，默认的events队列用schedule_work和schedule_delayed_work，而给定的用，

int queue_work(struct workqueue_struct *wq, struct work_struct *work);

int queue_delayed_work(struct workqueue_struct *wq,
					   struct work_struct *work,
					   unsigned long delay)

同样给定的队列刷新也有专门的，

flush_workqueue(struct workqueue_struct *wq);

下半部机制的选择

在2.6版本中，下半部机制有三种选择：软中断、tasklet、工作队列。 tasklet基于软中断，所以二者相近，而工作队列是靠内核线程实现。

软中断： 提供的保障最少，软中断处理函数必须格外小心共享数据的安全，可以在不同处理器上同时执行。代码多线索化的工作做得非常好则可以用，比如网络子系统，它完全使用单处理器变量。对于时间要求严格和执行频率高的任务也可以选择软中断。

tasklet： 接口简单，不并发运行，驱动程序尽可能选择tasklet。

工作队列： 在进程上下文中执行（提供重新调度和睡眠），开销最大，涉及到内核线程甚至上下文切换，每秒上千次的中断不建议用工作队列，如网络。

需要休眠就必须选择工作队列，否则最好用tasklet，追求高性能就用软中断。

在下半部之间加锁

在使用下半部机制时，即使在单处理器系统上，避免共享数据被同时访问也是至关重要。
tasklet，同一类型tasklet不允许同时执行，在不同的处理器上也不行。而不同类型的tasklet共享数据还是要使用锁机制。

进程上下文和下半部共享数据：访问数据之前，需要禁止下半部并得到锁，为了本地和SMP的保护并且防止死锁的出现。

中断上下文和下半部共享数据：访问数据前，需要禁止中断并得到锁，为了本地和SMP的保护并且防止死锁的出现。

任何在工作队列中的共享数据都需要锁机制，这个锁与内核中的锁没有区别，因为工作队列本身就在进程上下文中执行。

禁止下半部

一般单纯禁止下半部不够，常用做法是先获取锁再禁止下半部，驱动中就是这样干的。
禁止所有软中断和所有tasklet，用local_bh_disable()，允许执行则用local_bh_enable()。函数内部通过preempt_count（内核抢占时用的计数器）为每个进程维护一个计数器，当计数器为0时下半部才会被执行。
当然这些函数不能禁止工作队列。
